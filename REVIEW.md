# Code Review Guidelines

Правила ревью для проекта Expense Tracker. Применяются ко всем PR в ветку `main`.

---

## Общие правила

### TypeScript
- Запрещён `any` — использовать конкретные типы или `unknown` с narrowing
- Все публичные функции/методы должны иметь явные типы возврата
- `interface` для форм объектов, `type` для union/intersection/алиасов
- Общие типы между api и web — только через `@expense-tracker/shared`

### Именование
- Файлы: `kebab-case` (кроме React-компонентов — `PascalCase.tsx`)
- Переменные, функции: `camelCase`; классы, типы, интерфейсы: `PascalCase`; константы: `UPPER_SNAKE_CASE`
- Булевы переменные с префиксом: `is`, `has`, `can`, `should`

### Комментарии
- Не описывают «что» — только «почему», если это неочевидно
- Нет закомментированного кода в PR

### Безопасность
- Пользовательский ввод валидируется на границе системы (DTO для api, zod-схемы для web)
- Секреты только в `.env` — никогда не хардкодить, не логировать
- Все защищённые эндпоинты используют `@UseGuards(JwtAuthGuard)`
- Запросы к БД всегда фильтруются по `userId` текущего пользователя

## Пропускать при ревью
- Файлы миграций Prisma (`prisma/migrations/**`)
- `package-lock.json` и другие lock-файлы
- `*.log` файлы

## Backend (apps/api)

### Архитектура модуля
- Структура модуля: `commands/`, `queries/`, `dto/`, `entities/`, `*.repository.ts`, `*.module.ts`, `*.controller.ts`
- Handler не обращается к Prisma напрямую — только через Repository
- Controller не содержит бизнес-логику — только диспетчеризация через `CommandBus`/`QueryBus`
- Новый модуль регистрируется в `AppModule`

### CQRS
- Каждая операция — отдельная пара `*.command.ts` / `*.handler.ts` или `*.query.ts` / `*.handler.ts`
- Handler содержит всю логику: проверки, вызов репозитория, маппинг в DTO ответа
- Command/Query — простой класс-контейнер данных без логики

### DTO и валидация
- Все входящие DTO используют декораторы `class-validator` (`@IsString`, `@IsEnum`, и т.д.)
- Опциональные поля помечены `@IsOptional()`
- `whitelist: true` отсекает лишние поля — не нужно вручную игнорировать неизвестные свойства
- DTO ответа (`*-response.dto.ts`) не содержат `passwordHash` и других чувствительных полей

### Prisma и БД
- После изменения `schema.prisma` — обязательны `prisma:migrate` + `prisma:generate`
- `amount` хранится как `Decimal` — не конвертировать в `float` без явной необходимости
- Каскадное удаление настроено через схему (`onDelete: Cascade`), не через ручной код
- Отсутствует N+1: при связанных сущностях использовать `include` или `select`

### HTTP
- GET — без side-effects
- POST — создание ресурса, возвращает созданный объект
- PATCH — частичное обновление (не PUT)
- DELETE — возвращает `204 No Content` или удалённый объект
- Коды ошибок: `400` — невалидные данные, `401` — не аутентифицирован, `403` — нет прав, `404` — ресурс не найден

## Frontend (apps/web)

### FSD архитектура
Импорты строго по направлению: `app` → `pages` → `widgets` → `features` → `entities` → `shared`
- Слой не импортирует из вышестоящего слоя
- Публичное API слоя — только через `index.ts` (barrel-файл)
- shadcn/ui компоненты живут в `src/components/ui/` — не переносить в FSD-слои

### React-компоненты
- Один компонент — один файл; имя файла совпадает с именем компонента
- Пропсы типизируются через `interface Props` рядом с компонентом
- Нет inline-обработчиков сложнее одного выражения — выносить в переменную
- `useEffect` с зависимостями — зависимости перечислены полностью; пустой массив обоснован

### Next.js
- Server Components по умолчанию; `'use client'` только когда нужны хуки или браузерные API
- Переменные окружения для клиента — только через `NEXT_PUBLIC_` префикс
- Маршруты через файловую систему App Router (`app/` директория)

### Работа с API
- HTTP-клиент централизован в `shared/api/client.ts` — не создавать fetch/axios в компонентах напрямую
- Типы запросов/ответов — из `@expense-tracker/shared` или `shared/api/types.ts`
- Ошибки API обрабатываются на уровне feature, не в UI-компонентах

### Стилизация
- Только Tailwind CSS-классы; нет inline-стилей и отдельных CSS-файлов (кроме `globals.css`)
- Для условных классов использовать утилиту `cn()` из `lib/utils.ts`
- Нет магических значений цветов/размеров вне дизайн-токенов Tailwind

## Коммиты и PR

### Коммиты
- Формат Conventional Commits: `<type>(<scope>): <описание>`
- Описание на русском, строчная буква, без точки в конце
- Один коммит — одна логическая единица изменений
- Не коммитить `.env`, артефакты сборки, `node_modules`

### Pull Request
- Один PR = одна фича или один фикс
- PR описание содержит: Summary, реализованный функционал, технические детали, Test plan
- `pnpm lint && pnpm format:check` проходит без ошибок перед созданием PR
- Нет изменений, не связанных с задачей PR (случайный рефакторинг, правка отступов в чужих файлах)

### Чеклист перед аппрувом
- [ ] TypeScript компилируется без ошибок (`pnpm build`)
- [ ] Lint и форматирование чисты
- [ ] Нет `console.log`/`debugger` в коде
- [ ] Новые эндпоинты защищены `JwtAuthGuard` (если требуется)
- [ ] Изменения схемы БД сопровождаются миграцией
- [ ] Общие типы вынесены в `packages/shared`
